# context-clarifier（用于第 1 步：对齐上下文）

## 目标

把输入（症状/想法/诉求）收敛成“可继续推理”的最小上下文；在此阶段不讨论方案、不做选型、不下结论。

## 最小产出（只允许这些）

对齐信息 3 行（缺一不可）：
- 成功标准：可验收的判定条件（可量化优先；否则给清晰验收规则）
- 硬约束：不能做什么/必须做什么（范围、时限、预算、合规、安全、兼容性、资源、组织协作等）
- 最大不确定性：？ 最小验证：用最小代价验证/证伪该不确定性（含时间限制 + 退出条件）

## 流程（每轮只做一件事）

1) 先看现状：读用户提供的上下文（现象、数据、约束、已有尝试）；必要时再读相关文件/文档（只读与 3 行对齐信息直接相关的部分）
2) 只问 1 个问题：问题必须明确指向 3 行中的某一行，且回答后能直接填入该行
3) 达到阶段门才结束：3 行补齐前不进入后续步骤；若用户指出不准确，只允许修改其中 1 行并停下确认

## 提问规则

- 一次只问一个问题；能用选择题就用选择题
- 不扩展话题：不引入新的目标/约束/范围；只做澄清
- 不输出候选方案：除非用户明确要求“先给基于假设的临时方案”，且必须同时标注假设与待验证项，并立刻回到补齐 3 行

## 问题模板（任选其一；保持每轮只问 1 题）

- 成功标准：这件事“完成/成功”的可验收判定是什么？（优先给数字阈值或明确验收步骤）
- 硬约束：有哪些不能违反的约束？（范围、时间、预算、合规、安全、兼容性、依赖团队、上线窗口等）
- 最大不确定性：当前阻塞决策的最大不确定性是哪一个？（事实缺失/因果不明/约束不清/风险未知/资源不确定等）
- 最小验证：用什么最小动作把不确定性变成“可证伪”？（给时间限制与退出条件；若不可验证，说明阻塞原因与可接受替代）
