# 红旗清单（RF-*）

使用方式：仅当命中红旗或用户明确追问红旗时，才查本表；平时不要展开。

记号约定：`RF-x -> 1/2/5` 表示必须回退到第 `1/2/5` 步。

## 阻断红旗（必须回退）

- `RF-1 -> 1` 成功标准不可验收（无指标/阈值/验收方式）（例如：只说“更快/更稳”）
- `RF-2 -> 1` 硬约束缺失或冲突（例如：既“不能停机”又“必须全量重建”）
- `RF-3 -> 1` 最大不确定性/最小验证缺失或泛化（例如：最小验证写“需要更多信息”）
- `RF-4 -> 1` 最小验证不可执行且无替代路径（例如：无权限/无数据且无法加观测）
- `RF-5 -> 1` 用户未选方案却进入第 6/7 步（例如：未选 A/B/C 就给终稿/计划）
- `RF-6 -> 1` 用户否定对齐信息但未回退修正（例如：用户纠正硬约束但仍按旧约束推导）
- `RF-7 -> 2` 关键证据与关键假设冲突仍沿用（例如：证据否定根因但仍按该根因推荐）
- `RF-8 -> 2` 关键因果链断裂（收益机制无法自洽/无证据支撑）（例如：宣称“动作 X 必然提升指标 Y”但无因果/证据）
- `RF-9 -> 5` 致命漏洞且无可接受降级/隔离/回滚（例如：可能造成不可逆数据错误且无回滚）

## 可继续红旗（用户可选回退或接受风险继续）

- `RF-10 -> 1` 成功标准过粗（可验收但不足以指导权衡）（例如：只有“更稳定”）
- `RF-11 -> 1` 验证不够“最小”（过大/过慢/依赖多方）（例如：要跨团队一周才拿到信号）
- `RF-12 -> 1/2` 倾向“先改再看”（仅在有护栏+回滚+触发信号时可继续）（例如：先上线再观察指标）
- `RF-13 -> 2` 低概率高风险未被管理（至少补齐触发信号 + 护栏动作）（例如：想到数据错乱但没开关/隔离）
- `RF-14 -> 5` 权衡前提变化导致排序可能改变（例如：时限/资源变化导致取舍翻转）
- `RF-15 -> 5` 方案适用边界被推翻（例如：关键前提“不改接口”不成立）
